Feature-Driven Development
Introduction (History)

FDD was initially devised by Jeff De Luca to meet the specific needs of a 15-month, 50-person software development project at a large Singapore bank in 1997. This resulted in a set of five processes that covered the development of an overall model and the listing, planning, design,and building of features. The first process is heavily influenced by Peter Coad's approach to object modeling. The second process incorporates Coad's ideas of using a feature list to manage functional requirements and development tasks. The other processes are a result of Jeff De Luca's experience. There have been several implementations of FDD since its successful use on the Singapore project.

Objectives

The primary objective of FDD is to provide a structured and iterative framework for software development, focusing on the rapid and continuous delivery of specific features, thereby ensuring constant value addition for users

Klob's Method for Feature-Driven Development What?
  • What is FDD?

    • FDD (Feature-Driven Development) is an agile methodology focused on developing specific features of a software. It emphasizes planning, designing, and building distinct features, enabling frequent and incremental deliveries.

  • What are the steps of FDD?

    • Develop an overall model

    • Build a feature list

    • Plan by feature

    • Design by feature

    • Build by feature

How?
  • How is FDD implemented?
    1. Develop an Overall Model:
      • Create a global model of the application.
      • Team modeling sessions to establish a shared understanding of the system's structure and main components.
    2. Build a Feature List:
      • Break down the model into a list of small, manageable features.
      • Features should be concise, typically described in one sentence following the "action-result-object" format.
    3. Plan by Feature:
      • Prioritize and plan the development of features based on project needs and constraints.
      • Allocate resources and schedule development for each feature
    4. Design by Feature:
      • For each feature, create a detailed design before development.
      • Small group design sessions to work out the technical details of the feature.
    5. Build by Feature:
      • Develop, test, and integrate each feature.
      • Use coding practices, code reviews, and unit testing to ensure quality.
  • How to ensure quality in FDD?
    • Use automated testing, continuous integration, regular code reviews, and constant user feedback to ensure each feature meets quality requirements.
Why?
  • What are the steps of FDD?

    • Improved User Experience: By focusing on specific features, FDD ensures that critical aspects of the project are always prioritized.

    • Visibility and Control: Small iterations and frequent deliveries provide continuous visibility of project progress and early problem detection.

    • Adaptability: The iterative and incremental nature of FDD allows for easy adaptation to changing project needs or priorities.

    • Increased Collaboration: FDD encourages close collaboration among team members, ensuring better communication and a shared understanding of project goals.

What If?
  • What if we encounter technical issues?
    • Have a contingency plan that includes additional resources, alternative technical approaches, and expert consultations.
  • What if the feature does not meet user expectations?
    • Gather user feedback, prioritize fixes and improvements, and iteratively refine the feature.
  • What if we identify a better method or technology during the project?
    • Stay open to new technologies and methodologies, evaluate them against project goals and constraints, and adapt the FDD method accordingly if it can provide significant benefits.
  • Focus:
    The feature-centric approach ensures that critical aspects of the project are always prioritized.

  • Visibility:
    Small iterations and frequent deliveries offer continuous visibility of project progress.

  • Adaptability:
    The ability to adapt to changes is inherent in the iterative and incremental process.

  • Increased Collaboration:
    FDD encourages close collaboration among team members, ensuring better communication and a shared understanding of project goals.
  • Initial Modeling Complexity:
    Creating an overall model at the beginning can be complex and time-consuming.

  • Feature Overload Risk:
    Focusing on specific features can lead to feature overload, increasing system complexity.

  • Resource Intensive:
    FDD can require significant resources for modeling, planning, and detailed design of each feature.
Roles
  • Project Manager:
    Oversees the entire project and ensures communication with stakeholders.
  • Domain Architect:
    Responsible for initial modeling and technical coherence of the project.
  • Team Leader:
    Guides the development team in realizing features.
  • Developer:
    Designs and codes the features.
  • Quality Manager:
    Ensures each feature meets quality criteria.
Life Cycle
  • Develop Overall Model:
    Initial system modeling.
  • Feature List:
    Decompose the model into features.
  • Planning:
    Prioritize and plan features.
  • Design:
    Detailed design of features.
  • Construction:
    Develop, test, and integrate features
